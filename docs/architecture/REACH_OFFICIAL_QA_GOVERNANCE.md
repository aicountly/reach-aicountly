# Reach Official Q&A Governance — Phase 5

## Overview

This document defines the governance rules, ethical constraints, approval matrix, disclosure policy, and quality controls for official Q&A content produced by AICOUNTLY Reach.

---

## Core Ethical Rules

### No Fake Community Activity

The system never generates or stores fabricated:
- Users, profiles, or community members
- Questions presented as user-submitted when they are not
- Answers that impersonate independent community members
- Votes, helpful counts, view counts, follower counts
- Engagement history or fabricated timestamps

Official questions created by Reach are explicitly marked `source_type = 'official_question'` and are never misrepresented as user-generated.

### No AI Impersonation

AI-generated content may only be used as a draft for an authorised official AICOUNTLY response. AI content must never be presented as:
- Written by a customer or community member
- Written by a Chartered Accountant or professional advisor
- Written by a regulator or government authority
- Written by an independent expert

The `ai_assisted` flag is mandatory when AI materially contributed to the final published answer.

### Human Approval Requirement

**No AI-generated or AI-modified official answer may become public without an explicit human approval event.**

- Automated generation creates drafts only.
- Scheduling may release only a previously approved and checksummed version.
- Any content edit after approval invalidates the approval and requires reapproval.
- Approval is bound to a SHA-256 checksum of the version content.

### Grounding Requirement

Official answers must be grounded in approved AICOUNTLY sources. Unsupported claims block approval.

Grounding must be traceable to one or more of:
- Approved knowledge-base articles
- Approved blog posts
- Approved product documentation
- Approved release notes
- Approved product feature records
- Approved marketing knowledge
- Approved policy documents
- Official authoritative external references

## Official Identity Governance

### Allowed Official Identities

Official identities are administratively managed in `reach_community_official_identities`. No customer or community account maps to an official identity.

Pre-configured official identities:

| Slug | Display Name | Department |
|------|-------------|------------|
| `aicountly-support` | AICOUNTLY Support | Customer Support |
| `aicountly-product` | AICOUNTLY Product Team | Product |
| `aicountly-compliance` | AICOUNTLY Compliance Team | Compliance |
| `aicountly-engineering` | AICOUNTLY Engineering | Engineering |
| `aicountly-official` | AICOUNTLY Official | General |

### Identity Management Rules

- Official identities may only be created, modified, or deactivated by users with `community.identity.manage` permission.
- Official identities cannot be linked to customer accounts.
- Official identity slugs must not resemble or be confused with user account slugs.
- Deactivated identities are preserved for historical audit; their published content remains attributed.

## Approval Matrix

### Standard Answers

| Role | Can Generate Draft | Can Review | Can Approve | Can Publish |
|------|------------------|------------|-------------|-------------|
| `content_manager` | Yes | Yes | No | No |
| `senior_editor` | Yes | Yes | Yes (non-high-risk) | No |
| `community_manager` | Yes | Yes | Yes (non-high-risk) | Yes |
| `compliance_officer` | No | Yes | Yes (all) | No |
| `super_admin` | Yes | Yes | Yes (all) | Yes |

### High-Risk Answers (risk_classification = 'high' or 'critical')

- Require review by `compliance_officer` or `senior_editor` in addition to standard approval.
- `requires_professional_review = true` answers are flagged for professional sign-off before approval.
- Self-approval is blocked where the approver is the same user who generated/edited the draft.

### Separation of Duties

For answers generated by a named user:
- That user cannot approve the same answer (enforced in `OfficialAnswerApprovalService`).
- An override requires `super_admin` role with a mandatory reason.

## Disclosure Policy

Every official public answer must carry machine-readable metadata and visible text sufficient to establish:

### Required Metadata Fields

| Field | Description |
|-------|-------------|
| `is_official_answer` | TRUE for all Reach-published official answers |
| `official_identity_slug` | Slug of the official AICOUNTLY identity |
| `official_identity_name` | Display name for rendering |
| `ai_assisted` | TRUE if AI materially assisted drafting |
| `human_reviewed` | TRUE when a human reviewer explicitly approved |
| `approved_at` | ISO 8601 approval timestamp |
| `answer_version` | Current version number |
| `correction_note` | Non-null when answer has been publicly corrected |
| `withdrawn_at` | Non-null when answer has been withdrawn |

### Visible Disclosure Requirements

All public-facing official answers must display:
1. Official identity badge (AICOUNTLY identity name + department)
2. "AI-assisted draft" label when `ai_assisted = true`
3. "Reviewed by AICOUNTLY team" label when `human_reviewed = true`
4. Correction notice (styled banner) when `correction_note` is set
5. Withdrawn state notice when `withdrawn_at` is set

### Professional Advice Limitation

When `requires_professional_review = true` or `risk_classification` is `high`:
- A standard limitation notice is appended: "This answer provides general guidance only. For advice specific to your circumstances, consult a qualified professional."
- This notice cannot be suppressed without `super_admin` override with recorded reason.

## Grounding and Source Coverage

### Source Coverage Checks

Before approval, `OfficialAnswerValidationService` verifies:
- Each material factual claim has at least one approved source reference
- No claimed AICOUNTLY feature is absent from approved product feature records
- No compliance assertion lacks an approved source
- No source in the grounding snapshot has been withdrawn or expired

### Conflict Detection

Approval is blocked when:
- The answer conflicts with a newer approved source
- Grounding sources have expired
- Source conflicts remain unresolved

### Grounding Snapshot Retention

The exact source versions used at generation and approval time are stored in `reach_community_source_coverage`. These records are immutable and preserved for audit.

## Moderation Finding Categories

Moderation findings are stored in `reach_community_moderation_findings` and may block or require review before approval:

### Auto-blocking Findings

| Category | Action |
|----------|--------|
| `prompt_injection` | Block and flag for security review |
| `malicious_html` | Block and sanitise |
| `unsafe_links` | Block and flag |
| `personal_data` | Block unless operator explicitly reviews |
| `confidential_information` | Block |

### Review-Required Findings

| Category | Action |
|----------|--------|
| `legal_risk` | Route to professional_review |
| `tax_risk` | Route to compliance_officer |
| `unsupported_claims` | Flag for grounding review |
| `hallucinated_features` | Block until product fact check passes |
| `outdated_product_behaviour` | Flag for editorial review |
| `impersonation_risk` | Block and flag |

### Overridable Findings

Overrides require the `community.answer.override_validation` permission and a mandatory written reason, recorded in the audit log.

## Correction and Withdrawal

### Correction Process

1. A correction is triggered by: source supersession, user report, verification mismatch, or operator action.
2. `OfficialAnswerCorrectionService` creates a new version marked `correction_draft`.
3. The correction draft goes through the standard editorial/approval workflow.
4. On approval, the correction is published with a visible `correction_note`.
5. Historical versions are preserved with `superseded_by` references.
6. A `community.answer.corrected` audit event is recorded.

### Withdrawal Process

1. Withdrawal removes content from public view.
2. The content record is preserved with `withdrawn_at` timestamp.
3. The public site marks the answer with `status = 'withdrawn'` and renders a placeholder.
4. The canonical URL returns a 410 Gone or renders a withdrawal notice.
5. All audit history is preserved.
6. Content can be restored if the withdrawal reason is resolved.

### Prohibition on Silent Rewrites

A published answer must never be silently overwritten. Every change creates a new version. Every material edit invalidates approval.
